# Development

## Environment

Development and packaging target Windows x64 with the `x86_64-pc-windows-msvc` toolchain.

Common local entry points:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\run.ps1 debug
powershell -ExecutionPolicy Bypass -File .\scripts\run.ps1 pipe -s 1 --device-info
```

`scripts/run.ps1` is only for development and local builds. End users use the release zip or the release `install.ps1` asset.

## Packaging

The release package is generated by `scripts/package.ps1`:

```powershell
.\scripts\package.ps1 -Version v0.7.0 -TargetDir target
```

The package currently contains:

- `winmon.exe`
- `LICENSE`
- `README.txt`
- `third_party/licenses/*`

`OpenHardwareMonitor` is no longer shipped as an external exe. On first launch, `winmon.exe` writes the embedded `OpenHardwareMonitorLib.dll` into `%APPDATA%\winmon\third_party\ohm`.

## Bootstrap

On startup, the program bootstraps itself by:

- copying the current `winmon.exe` to `%APPDATA%\winmon\winmon.exe`
- writing the embedded `OpenHardwareMonitorLib.dll` to `%APPDATA%\winmon\third_party\ohm`
- appending `%APPDATA%\winmon` to the user `PATH`

The installer and release flow depend on this behavior.

`scripts/install.ps1` prefers `curl.exe` for release downloads to avoid unreliable `Invoke-WebRequest` behavior on Windows PowerShell 5.1.

## Release

The release flow has two parts:

1. push to a normal branch to trigger `check`
2. push a `v*` tag to trigger `release`

If asset names or download URLs change, update these together:

- `.github/workflows/release.yml`
- `scripts/install.ps1`
- `scripts/package.ps1`

## example-grafana

`example-grafana` is currently wired to scrape the Windows host directly on the LAN:

- `192.168.8.16:9090`

If the `winmon serve` host or port changes, only update `example-grafana/prometheus.yml`.

If Docker is available on the local machine:

```bash
cd example-grafana
docker compose up -d
```

The current dashboard only keeps stable `winmon` metrics. `sys_power` is no longer pursued and stays `N/A`.

## License

Repository code is released under `MIT`.

Keep the scope clear:

- root `LICENSE` covers `winmon` itself
- `third_party/licenses` keeps original upstream and dependency license texts

## winget

The repository already contains `winget` manifest files under `winget/manifests/a/Ailuntz/Winmon/`.

For a new release:

```powershell
.\scripts\gen-winget.ps1 -Version 0.7.0 -InstallerSha256 <sha256>
```

Notes:

- installer URLs must be publicly accessible before submitting to `microsoft/winget-pkgs`
- the manifest schema should follow current community requirements, currently `1.12.0`
